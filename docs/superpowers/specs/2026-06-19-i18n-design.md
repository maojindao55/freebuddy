# FreeBuddy i18n Design

**Date:** 2026-06-19
**Status:** Approved
**Scope:** Add internationalization (English + 简体中文) to the FreeBuddy desktop app.

## Context

FreeBuddy is an Electron + React 19 + Vite + TypeScript desktop AI workspace using Zustand
and better-sqlite3. Prior to this work it has **zero i18n infrastructure**: no i18n library,
no locale files, no translation functions, and no place to persist a language preference
(the theme toggle isn't even persisted today). The UI is a mix of ~70 English strings and
~30 Chinese strings scattered inline across the renderer (~140–160 distinct strings total),
and `index.html` declares `lang="zh-CN"` despite most of the UI being English.

## Goals

- Support **two locales**: `en` (source/default) and `zh-CN`.
- Translate **all user-facing strings**: UI labels/buttons, errors/toasts, placeholders,
  tooltips/aria, Electron menus/dialogs, and the semi-technical runtime + AI-prompt strings.
- **Auto-detect** the OS locale on first run, with a **manual override** in Settings that
  persists across restarts.
- Keep the Electron menu's built-in `role:` items auto-localized by the OS (no change).
- Persist theme is **not** part of this scope (language-only).

## Non-Goals

- No additional languages beyond English and Simplified Chinese.
- No server-side/remote translation loading (resources bundled at build time).
- No component-render tests (the project has no React test harness today).
- No translation of dev-facing `throw` messages that never reach the UI.

## Decisions

| Decision | Choice |
|---|---|
| i18n library | `react-i18next` + `i18next` (core) |
| Source/default language | `en` |
| Second language | `zh-CN` |
| Language selection | Auto-detect on first run + manual override in Settings, persisted |
| Translation scope | Everything user-facing |
| Resource format | JSON, bundled at build time |
| Theme persistence | Out of scope |

## Architecture

### Components

1. **i18n layer (renderer)** — `src/i18n/index.ts`
   - Initializes `i18next` + wires `react-i18next`.
   - Loads `en` and `zh-CN` resource bundles.
   - `fallbackLng: 'en'`; initial `lng` read from the settings store (or `navigator.language`
     on first run, before a choice is persisted).
   - Imported once in `src/main.tsx` before `<App />` mounts.

2. **i18n layer (Electron main)** — `src/i18n/main.ts`
   - Initializes **core `i18next`** (non-React) with the **same shared JSON** resources.
   - Language read from the `app_settings` table at startup.
   - Covers: custom menu labels, HTTP error body in `electron/main.ts`, dialog filter names in
     `electron/cli/ipc.ts`.
   - The menu is rebuilt when the language changes.

3. **Settings store + persistence (NEW)** — `src/store/settingsStore.ts`
   - Zustand store holding `language`.
   - On init: loads the persisted language from main (or triggers first-run detection).
   - On change: calls `i18next.changeLanguage(lng)` (renderer) and persists to SQLite via IPC.
   - Mirrors the existing SQLite/IPC/Zustand pattern used for CLI adapters.

4. **Persistence backend**
   - **SQLite table** `app_settings(key TEXT PRIMARY KEY, value TEXT)` — added to the startup
     migration in `electron/cli/db.ts`.
   - **IPC handlers** `settings:get` / `settings:set` in `electron/cli/ipc.ts`.
   - **Preload** exposure in `electron/preload.ts` as `window.freebuddy.settings.get/set`.

5. **Settings UI** — `SettingsModal` gains a **"General"** section with a Language selector
   (English / 简体中文). Changing it updates the store → `i18next.changeLanguage()` → persists.
   `document.documentElement.lang` is updated reactively for a11y correctness.

### Data Flow

```
First run:
  navigator.language / app.getLocale()
    -> detect(): "zh*" => "zh-CN", else "en"
    -> settingsStore.init() -> persist to app_settings

Subsequent runs:
  app_settings.language
    -> settingsStore.init()
    -> i18next.changeLanguage(lng)
    -> <App /> renders in lng

Manual override:
  Settings "General" -> Language selector
    -> settingsStore.setLanguage(lng)
    -> i18next.changeLanguage(lng)  (renderer)
    -> IPC settings:set { language: lng }
    -> app_settings upsert
    -> (main) re-init core i18next + rebuild menu
```

### Key Structure & Resources

- Files: `src/locales/en.json`, `src/locales/zh-CN.json`.
- Nested by feature area, deduping repeated strings:
  - `common.*` — `cancel`, `close`, `settings`, `delete`, `save`, `reset`, …
  - `chat.*` — `inputPlaceholder`, `attachFile`, `send`, `stop`, `permission.*`, starter prompts, …
  - `workspace.*` — `activeAgent`, `runState`, `messages`, `agentTurns`, `context`, `cost`, statuses, …
  - `conversations.*` — `title`, `new`, `empty`, `deleteConfirm`
  - `settings.*` — `general.language`, `cli.title`, `cli.*` labels, …
  - `errors.*` — `taskFailed`, `sendFailed`, `checkFailed`, `cliBridgeUnavailable`, `agentNotInstalled`, …
  - `permission.*` — `title`, `input`, `help`, `morePending` (plural)
  - `status.*` — `running`, `starting`, `done`, `failed`, `killed`
- Interpolation: `t('errors.taskFailed', { err })` → `"Task failed: {{err}}"`.
- Plural: `permission.morePending` with `one`/`other` forms for `+N more pending request(s)`.

## Migration Details (~140–160 strings)

All inline strings become keys. Chinese strings get a new English source value. Notable
restructures beyond simple key substitution:

- **`StreamItem.tsx:289-296`** action verbs ("读取/修改/执行 {{target}}") → keys with the verb
  passed as a parameter (e.g. `tool.read`/`tool.edit`/`tool.exec`).
- **`ChatView.tsx:19-28`** starter prompts → array of keys (`chat.starter.*`).
- **`chatAttachments.ts`** AI-prompt template strings → keys (still injected into the prompt,
  but localizable).
- **`message.status`** raw values at `MessageBubble.tsx:423`, `WorkspacePanel.tsx:84` →
  `status.*` translations.
- **`ConversationList.tsx`** `Intl.DateTimeFormat(undefined,…)` → wired to current language.
- **`index.html`** `lang="zh-CN"` → set dynamically at runtime; `<title>` left as "FreeBuddy".
- **`AvatarPicker.tsx`** `GROUPS` array → keys (`settings.avatar.groups.*`).
- **`config/aiMembers.ts`** `description` and **`config/cliAdapters.ts`** `label` fields → keys.

## Testing

The project uses `node --test tests/*.mjs`. There is no React/component test harness, so
component-render tests are out of scope. New node tests cover:

- **Settings IPC round-trip** — `settings:set` then `settings:get` returns the value, persisted
  to SQLite (in-memory or temp DB, mirroring existing test patterns).
- **Locale-detection pure function** — `detectLocale('zh-CN')` ⇒ `'zh-CN'`,
  `detectLocale('en-US')` ⇒ `'en'`, `detectLocale('zh-TW')` ⇒ `'zh-CN'`, unknown ⇒ `'en'`.
- **Interpolation correctness** — a thin check that a sample key with `{{err}}` renders the
  substituted value for both locales.

Manual QA matrix:
1. Fresh launch (no stored language) → auto-detect picks based on OS locale; UI matches.
2. Settings → switch to 简体中文 → every screen (Chat, Workspace, Settings, Permission dialog,
   Electron menu, file dialog filters) updates immediately.
3. Restart app → language choice persists.

## Risks & Mitigations

- **Main/renderer language drift.** Both read from the same `app_settings` value and the same
  JSON resources; the override flow updates both atomically.
- **Forgotten strings.** Mitigated by key-by-file migration checklist in the implementation
  plan and the manual QA matrix.
- **Status enum leakage.** Raw `running`/`done`/… values currently rendered directly are routed
  through `status.*` keys so they translate without breaking the underlying enum.
- **Bundle size.** Two small JSON locale files; negligible.
