# 新建业务空间页面 UI/UX 优化设计

- 日期：2026-06-28
- 范围：「新建 / 编辑业务空间」页面（`BusinessWorkspaceEditor`）
- 类型：前端 UI/UX 重构，无后端/数据模型变更

## 背景与目标

「新建业务空间」是用户把多个代码仓库组织成一个业务、并为每个仓库分配 Agent 的入口。当前页面功能可用，但体验粗糙：

- 一整页长表单（名称 → 模板 → 仓库列表 → 协作 → 策略）一次性铺开，新用户无从下手。
- 复用通用的 `workflow-team-editor` 样式，没有为「业务空间」定制视觉。
- 点击模板会**静默覆盖**已填的仓库，没有选中态、没有二次确认。
- 仓库「端类型」下拉显示原始英文值（`client`/`server`/`admin`…），与全中文界面不一致；类型徽章也是裸英文。
- 目录选择按钮只是一个 `…`，含义不明。
- 校验只在顶部弹一句原始错误（缺名称时甚至把字段名当错误信息），无内联反馈。

目标：系统性地提升该页面的**结构引导感、视觉专业度、细节体验**，让「新建」这个动作清晰、可信、不易出错。

## 方案选择

对比了三种交互模型：

- **A · 引导式向导**：分 4 步 + 顶部进度条。认知负担最低，但老手配多仓库时强制分步、多点几次。
- **B · 精简单页**：仍一页，加吸顶锚点导航 + 分节卡片。效率高，但首屏仍是「一长串表单」。
- **C · 模板优先混合（已选）**：以模板大卡为入口，选中即预填并展开单页编辑；高级默认折叠。

**选择 C**：用「先选模板」这个动作把零散字段串成一条主线，既给新用户明确的起点，又保留单页编辑的效率，避免向导的强制分步。

## 详细设计

### 整体流程（单页，自上而下）

1. **从模板开始**（hero 区，4 张大卡，始终可见）
2. **基础信息**（名称、描述）
3. **代码仓库**（按所选模板预填；可增删改）
4. **协作 & 提交策略**（单个折叠区，默认收起，带可调项数徽标）

顶部保留现有头部操作区（返回 / 取消 / 主操作按钮），不新增页脚；主操作按钮文案随新建/编辑切换（见下）。不再是「先填名称才出现模板」的顺序——模板前置，名称紧随其后。

### 模板入口区（hero）

- 4 张模板卡（C端+服务端+管理 / 前端+服务端 / 单仓 / 自定义），2 列网格。
- 每张卡显示：标题、简述、仓库数量（如「3 仓库」）。
- **选中态**：当前选中模板有品牌色描边（`--fb-brand`）+ 柔和外发光 + 右上角 ✓ 圆点。
- **切换确认**：当用户已存在仓库（`draft.surfaces.length > 0`）且点击了**不同**模板时，弹 `window.confirm`（与现有删除确认一致的轻量模式）提示「切换模板会替换当前仓库列表，是否继续？」。`draft.surfaces` 为空或点同一模板则直接应用，不弹窗。
- 编辑已有 workspace 时，进入时根据现有 surfaces 做尽力匹配：若与某模板的 surface 集合一致则高亮该模板，否则不高亮（视为自定义）。

### 基础信息

- 名称：标红「*」必填标记；保存时若为空，输入框边框变红 + 下方红色提示「请填写名称」，并滚动定位到该字段。不再把字段名当错误文案塞进顶部错误列表。
- 描述：保持多行文本框。
- 顶部仍保留返回 / 取消 / 主操作按钮区，主按钮文案在「新建」时为「创建业务空间」，「编辑」时为「保存」。

### 代码仓库列表

- 每个仓库一张卡。卡片头部：**端类型徽章（汉化 + 图标）** + 仓库名称。
- 端类型徽章汉化与配色：
  - `client` → ◐ C端（绿）
  - `server` → ● 服务端（蓝）
  - `admin` → ◑ 管理（琥珀）
  - `shared` → ◐ 共享
  - `docs` → ▤ 文档
  - `test` → ✓ 测试
  - `custom` → ⬚ 自定义
- 类型下拉同样显示汉化文案（`option` 文本走 i18n），值仍为原始英文 kind。
- 卡片内容栅格：仓库路径（输入框 + **「选择目录」**按钮）、默认 Agent。
  - 目录按钮由 `…` 改为 `t('business.chooseDirectory')` = 「选择目录」/ "Choose directory"。
- 「验证命令」「允许路径」「职责」「启用」收敛进卡片内的 `<details>`（仓库高级设置），保持现状的折叠交互。
- 删除按钮保留。

### 协作 & 提交策略（折叠合并）

- 把原「协作关系」section 与「高级设置（policy）」`<details>` **合并为单个折叠区**「协作 & 提交策略」，默认收起。
- 折叠标题右侧显示可调项数徽标（如「5 项可调」），数值 = 启用仓库数（协作角色）+ 3（policy 项），让用户知道收起的是什么而不产生负担。
- 展开后内部仍按现有字段渲染（每个仓库一个契约角色下拉 + policy 的 3 个开关 / 分支模板）。

### 校验与错误反馈

- 名称必填，内联红色反馈（见「基础信息」）。
- 保存失败（store 返回 `ok=false`）时，顶部错误列表保留，但文案改为可读的 `t('business.saveFailed')`，不再出现裸字符串 `"save failed"`。

## 视觉与样式

- 为业务空间编辑器新增/细化专属类，减少对 `workflow-team-editor-*` 的视觉复用（结构类可继续复用，但卡片、模板、徽章走 `business-*` 专属样式）。
- 模板卡选中态使用品牌色（`--fb-brand` / `rgba(16,185,129,.07)` 背景 + 外发光），与现有 `.business-workspace-template:hover` 的品牌色逻辑一致。
- 徽章按类型分色（绿/蓝/琥珀），使用低饱和背景 + 深色文字。
- 保持 DESIGN.md 的圆角（`--rounded-md/lg`）、间距（`--fb-*` 变量）与字体体系，不引入新设计语言。

## 文件改动范围

- `src/components/Settings/BusinessWorkspaceEditor.tsx`
  - 重排 JSX：模板 hero 前置；合并协作与策略为单个折叠区。
  - 新增 `selectedTemplateId` 状态 + 切换确认逻辑 + 模板匹配反推。
  - 名称必填的内联错误状态（`nameError` / `touched`）。
  - 类型下拉与徽章接入汉化 + 图标映射。
  - 目录按钮文案；主按钮文案随新建/编辑切换。
- `styles.css`
  - 新增模板选中态、徽章分色、hero 区、折叠项数徽标等样式；微调间距与层级。
- `src/locales/zh-CN.json`、`src/locales/en.json`
  - 新增 key：`business.chooseDirectory`、`business.saveFailed`、`business.nameRequired`、`business.switchTemplateConfirm`、`business.collaborationAndPolicy`、`business.advancedCountHint`（带 `{{count}}`）、`business.createWorkspace`、`business.kind_<kind>`（7 个）、模板卡的仓库数复用现有或新增 `business.templateRepoCount`。
  - 现有 key 尽量复用，避免冗余。

## 不在范围内（YAGNI）

- 不做多步向导（已选 C）。
- 不改数据模型 / store / service（`BusinessWorkspace` 结构不变）。
- 不改列表页（`BusinessWorkspaceList`）与设置 tab 外壳。
- 不引入图标库依赖；图标用 Unicode 字符（与徽章草案一致），零新增依赖。
- 不做实时预览/契约生成（那是运行时页面的事）。
- 不做暗色模式专项调色（沿用现有 CSS 变量驱动）。

## 验收标准

1. 进入「新建业务空间」时，首屏看到模板 hero（4 张卡），名称/描述紧随其后。
2. 点击模板立即预填仓库并显示选中态；仓库为空时切换不弹确认，已有仓库且切换到不同模板时弹确认。
3. 名称留空点保存 → 输入框红框 + 「请填写名称」，不创建。
4. 仓库类型徽章与下拉均为中文 + 图标；目录按钮为「选择目录」。
5. 「协作 & 提交策略」默认折叠，徽标显示可调项数；展开后字段与原先等价。
6. 新建主按钮文案为「创建业务空间」，编辑时为「保存」。
7. 保存失败显示可读错误，不出现裸 `save failed`。
8. `npm run typecheck` 通过；中英文 locale key 对齐。
